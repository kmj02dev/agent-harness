# LESSONS

Reflexion에서 축적된 교훈. 같은 실수를 반복하지 않기 위해 존재한다. 매 작업 전 읽는다.
**크기 관리**: 최대 20개 유지. 초과 시 가장 오래되거나 중복되는 항목 삭제.

## Architecture
- Low-level 에이전트 루프를 직접 구현하는 것은 과잉 엔지니어링이다. Claude Code가 이미 프로덕션급 루프를 갖고 있으며, Context Engineering으로 행동을 제어하는 것이 더 효과적이다.
- 모델 성능이 아닌 **하네스(인프라)**가 경쟁 우위를 결정한다 (Manus 사례).
- 가장 큰 성능 향상은 복잡한 것을 추가할 때가 아니라 **불필요한 것을 제거**할 때 발생한다.

## Context
- CLAUDE.md의 모든 줄이 행동을 바꿔야 한다. 행동을 바꾸지 않는 줄은 토큰 낭비다 (ETH Zurich: 잘못된 CLAUDE.md는 4%만 도움, 토큰 14-22% 낭비).
- 파일시스템이 곧 외부 메모리다 (Manus 패턴). 벡터 DB나 복잡한 메모리 시스템보다 TODO.md/LESSONS.md가 더 효과적이다.
- 컨텍스트는 가장 희소한 자원 — 탐색은 subagent에 위임하고 메인 컨텍스트를 보호한다.

## Agent Design
- 35분+ 작업 시 모든 에이전트에서 성능 열화가 발생한다. 장기 작업은 체크포인팅 또는 세션 분리가 필수.
- 멀티에이전트 4개 이상에서 조정 세금이 성능 이득을 초과한다. 2-3개가 최적.
- 단순한 루프 + 좋은 도구 인터페이스가 복잡한 멀티에이전트보다 안정적이다 (Claude Code, SWE-agent 사례).

## Mode Design
- 활동 모드(어떻게)와 도메인(무엇에 대해)을 분리해야 조합 폭발을 피한다.
- Teaching 모드는 유일하게 "되묻기 허용" — 자율 실행 원칙의 예외. 목표가 "사용자의 이해"이므로 상호작용이 필수.
- 새 모드를 추가할 때는 "이 모드가 없으면 에이전트가 어떤 실수를 하는가?"로 정당성을 판단한다. 실수를 막지 않는 모드는 불필요.

## Maintenance
- 하네스 파일들은 정기적으로 staleness 점검 필요. 기능이 추가될 때 WORKLOG/TODO/DECISIONS가 동기화되지 않으면 다음 세션에서 혼란 발생.
