# Teaching Mode Development — LESSONS

teaching 모드 개발 과정에서 나온 미검증 가설. 이 모드 재수정 시 로드한다.

- GAN 루프 4라운드면 모드 파일이 수렴한다 (teaching에서 1/6 → 6/7 달성).
  **검증**: 다른 모드 개발에서도 4라운드 내 수렴하는지 관찰. 확인 시 harness-dev Flow에 "4라운드 기준" 추가.
- 사용자의 질문 패턴이 모드 실패를 진단하는 신호다. "수식에서 어디?" = 수식-직관 매핑 부재, "X와 같은가?" = 기존 지식 연결 부재, "왜 이 값?" = 유도 과정 부재.
  **검증**: 다른 모드 개발에서도 사용자 피드백 패턴 → 규칙 결함 매핑이 가능한지 관찰. 확인 시 harness-dev Flow step 1에 "사용자 피드백 패턴 분석" 추가.
- 개념 이해와 구현 역량은 별개다. L1-L3을 모두 통과해도 구현 테스트에서 0%인 영역이 존재했다.
  **검증**: 다른 논문 교육에서도 같은 패턴(이해 OK, 구현 NOT OK) 관찰 시 확인. 확인 시 CLAUDE.teaching.md에 "이해 확인(L1-L3)과 구현 확인을 분리한다" 규칙 추가.
- 사자 단계에서 "모른다"의 원인은 분석 부족과 지식 부족 두 가지다. 한계 식별(3/3 자력)은 논문 내부 지식으로 가능했지만, 실험 비판(1/5)·기여 평가(0)는 분야 지식 부재가 원인이었다. 동일한 "모른다"에 다른 대응이 필요하다.
  **검증**: 다른 논문 교육에서도 Phase 1(논문 내부)과 Phase 2(분야 맥락)의 성공률 차이가 관찰되는지 확인. 확인 시 규칙 유지, 미관찰 시 규칙 재검토.
- 스캐폴딩(맥락 정보 제공) 후 사용자의 분석 성능이 급상승한다: 실험 비판 1→5, 기여 평가 0→3. 병목이 지식이면 정보 제공이 해결책이고, 분석이면 힌트가 해결책이다.
  **검증**: 다른 논문에서도 스캐폴딩 전후 동일한 성능 차이 관찰 시 확인. 확인 시 Phase 2 스캐폴딩을 필수 규칙으로 승격.
- 아이 단계는 사자 단계의 한계 식별 결과에 의존한다. 사자에서 좋은 한계를 찾을수록 아이에서 더 구체적인 아이디어가 나온다.
  **검증**: 다른 논문에서 사자 결과 품질과 아이 결과 품질의 상관 관찰. 확인 시 "사자 단계 한계 식별 최소 3개" 규칙 추가.
