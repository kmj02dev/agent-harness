# Survey Mode

연구 동향을 조사하고 구조화된 보고서로 정리하는 단계다. Research의 확장이다.

## Flow
1. 조사 범위 정의 — 주제, 기간, 포함/제외 기준
2. 분류 체계 수립 — Taxonomy 설계 원칙 참조
3. 탐색 — subagent 3-5개 병렬 실행 (영역별 분담)
   - 최소 3개: 조사 범위를 3개 이상 영역으로 분할
   - 최대 5개: 조정 세금 한계
   - 각 subagent에게 명확한 조사 범위 + 출력 파일명 지정
   - 각 subagent는 중간 보고서를 별도 .md 파일로 저장
4. 수집 결과를 분류 체계에 매핑
5. 비교표/타임라인/트렌드 맵 작성
6. **Synthesis 작성** — 트렌드별 서술 규칙 참조. 논문 요약 나열 금지.
7. Gap 분석 — Gap Analysis 규칙 참조
8. Quality Checklist 검증 후 보고서 작성 → 파일로 저장

## Taxonomy 설계 원칙
분류 체계는 survey의 핵심 기여다. 독자가 새 논문을 접했을 때 즉시 분류할 수 있어야 한다.
- **MECE 검증 필수**: 분류 축 설계 후 반드시 확인 — (1) 상호배제: 하나의 논문이 한 차원 내에서 하나의 범주에만 속하는가? (2) 전체포괄: 모든 조사 논문이 최소 하나의 범주에 포함되는가?
- **축 혼재 금지**: "무엇을(what)" 축과 "어떻게(how)" 축을 분리한다. 예: KB 구조(KG/NN/LLM)는 what, 동기화 기법(FL/Distillation)은 how. 하나의 분류 차원에 what과 how를 섞지 않는다.
- **반복 설계**: 첫 분류가 MECE를 만족하지 않으면 축을 재설계한다. 분류 불가 논문이 나오면 축을 확장하거나 범주를 재정의한다.

## 트렌드별 서술 규칙 (Synthesis)
Summary(논문 요약 나열)가 아니라 Synthesis(논문 간 관계 분석)를 작성한다.
각 트렌드에 다음 3가지를 필수 포함:
1. **발전 경로(Evolution)**: 시간순으로 연구가 어떻게 진화했는가. 예: "A(2022)가 X를 제시 → B(2024)가 Y로 확장 → C(2025)가 Z 한계를 해결"
2. **미해결 긴장(Unresolved Tensions)**: 연구 간 trade-off, 모순, 또는 논쟁. 예: "성능 vs. 해석가능성", "확장성 vs. 프라이버시"
3. **교차 분석(Cross-trend Links)**: 다른 트렌드와의 연결점. 예: "트렌드 1의 FL은 트렌드 4의 보안 위협에 노출됨"

나쁜 예: "A는 X를 했다. B는 Y를 했다. C는 Z를 했다."
좋은 예: "A가 X를 제시한 후, B는 Y로 확장했으나 Z 한계가 남았다. C가 이를 해결했지만, 이는 트렌드 3의 W와 trade-off 관계에 있다."

## Gap Analysis 규칙
단순 나열("X는 아직 연구되지 않았다") 금지. 각 open problem에 다음을 포함:
1. **Gap의 원인**: 왜 미해결인가? (기술적 난이도? 데이터 부족? 관심 부재?)
2. **기존 시도**: 어떤 접근이 있었고 왜 불충분한가?
3. **유망한 접근 방향**: 인접 분야(ML, 정보이론, 보안 등)에서 차용 가능한 기법 또는 논리적 다음 단계

## 비교표 규칙
- 비교표는 최소 1개 포함 — 텍스트보다 표가 동향 파악에 효과적
- **비교 축의 평가 기준을 정의한다**: "높음/중간/낮음" 사용 시 각주 또는 범례로 기준 명시. 예: "확장성: 높음(1000+ 에이전트), 중간(10-100), 낮음(<10)"
- 가능하면 정량적 성능 열(BLEU, accuracy, latency 등)을 포함한다
- 빈 셀이 과반이면 해당 축을 삭제하거나 재설계한다

## 커버리지 검증 (중간 → 최종)
최종 보고서 작성 후, 각 중간 보고서의 핵심 발견(Key Findings/Insights)이 최종 보고서에 반영되었는지 검증한다.
- 중간 보고서에서 3줄 이상 서술된 논문/기법이 최종 보고서에서 참고문헌에만 존재하면, 본문에 반영하거나 의도적 제외 사유를 기록한다.
- 중간 보고서의 비교표가 최종 보고서에서 사라졌으면, 통합하거나 제외 사유를 기록한다.

## 전망 규칙
- 전망은 반드시 [INFERENCE] 라벨 — 사실과 추측을 구분
- **다양한 관점**: 긍정적 발전 방향만 나열하지 않는다. 최소 1개는 해당 분야의 근본적 한계, 실패 가능성, 또는 대안 경로를 제시한다.

## 인용 규칙
- 모든 사실 주장에는 참고문헌 번호를 반드시 달아야 한다 — 출처 없는 사실 기술 금지.
- 인용 형식: "DeepSC [9]는 최초의 DL 기반 텍스트 SemCom이다."
- 추측/예측은 [INFERENCE] 라벨: "LLM 에이전트 기반 통신이 보편화될 것이다 [INFERENCE]."
- 본문의 번호와 참고문헌 섹션의 번호가 일치해야 한다.
- **서지정보 완결성**: 각 참고문헌에 저자, 제목, 출처, 연도, URL을 모두 포함한다. 저자 불명 시 "Authors unknown" 명시.

## Quality Checklist (보고서 완료 전 반드시 검증)
- [ ] 보고서 템플릿의 8개 섹션 모두 존재
- [ ] 참고문헌 30개 이상, 번호 참조 형식
- [ ] 본문 인용 번호와 참고문헌 번호 일치
- [ ] 비교표 최소 1개, 축 평가 기준 정의됨
- [ ] 트렌드 3-5개로 압축 (너무 많으면 "동향"이 아니라 "나열")
- [ ] 모든 사실 주장에 참고문헌 번호 인용 (출처 없는 사실 기술 없음)
- [ ] 추측에 [INFERENCE] 라벨
- [ ] Open Problems 섹션에 미해결 문제 3개 이상, 각각 원인+시도+방향 포함
- [ ] 전망 섹션 존재, 최소 1개 비판적/한계 관점 포함
- [ ] **분류 체계 MECE 검증 통과**
- [ ] **각 트렌드에 발전 경로 + 미해결 긴장 + 교차 분석 포함**
- [ ] **중간 보고서 커버리지 검증 통과** (핵심 발견 미반영 논문/비교표 없음)
- [ ] 참고문헌 서지정보 완결 (저자, 제목, 출처, 연도, URL)

체크리스트 미통과 시 "완료" 선언 금지. 부족한 항목을 보완 후 다시 검증.

## 보고서 구조 (기본 템플릿)
```
# [주제] 연구 동향 (기간)

## 1. 개요
- 조사 범위, 방법, 분류 체계

## 2. 핵심 트렌드 (3-5개)
- 트렌드별 설명 + 대표 연구/제품
- 각 트렌드: 발전 경로 + 미해결 긴장 + 교차 분석

## 3. 기술/모델 비교표
- 비교표 최소 1개, 성능/접근법/장단점
- 축 평가 기준 정의 포함

## 4. 주요 연구 그룹/기관

## 5. 표준화/산업 동향 (해당 시)

## 6. Open Problems
- 아직 해결되지 않은 문제 (각각 원인 + 기존 시도 + 유망 방향)

## 7. 전망
- 향후 방향 예측 ([INFERENCE] 라벨 필수)
- 최소 1개 비판적/한계 관점

## 8. 참고문헌
- 번호 참조 형식: [1], [2], ... 본문에서 인용 시 해당 번호 사용
- 각 항목: 번호, 저자, 제목, 출처, 연도, URL
- 최소 30개
```

## Rules
- Research 모드의 모든 규칙을 따른다 (증거 라벨링, 교차 확인, subagent 위임)
- 비교표는 최소 1개 포함 — 텍스트보다 표가 동향 파악에 효과적
- 전망은 반드시 [INFERENCE] 라벨 — 사실과 추측을 구분
